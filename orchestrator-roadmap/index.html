01 · Agents & Architecture

Six components, one narrow job each

The orchestrator is deliberately not where the intelligence lives — that's the domain agents' job. Its six components exist to make sure the right agent handles the right work, with the right level of trust, on the record.

01

Trigger Router / Ingress

Single entry point

The single entry point for everything the system reacts to. Four distinct trigger types arrive here: a scheduled job ticking over, an event arriving from a message bus, a monitored metric crossing a threshold, or an analyst making an on-demand request. The router authenticates the caller, normalizes whatever shape the payload arrives in into one canonical envelope, and classifies both what kind of work this is and how urgent it is — including whether it needs to bypass everything else via the real-time fast path.

Authentication & validationPayload normalizationIntent & priority classificationFast-path detection
02

Confidence Decision Engine

Trust scoring

Every proposed action a domain agent produces is scored here before it's allowed to do anything. The score isn't just the model's own self-assessment — it factors in how fresh and complete the underlying data was, how accurate this use case has historically been, and how far the result deviates from what a corrective-action playbook would expect. That number decides which of three tiers the action lands in: confidently correct enough to run on its own, good enough to recommend but not to trust blindly, or uncertain enough that a person needs to look first.

Evidence-weighted scoringThree-tier autonomy routingThreshold calibration over time
03

Agent & Tool Registry

Resolution & scope

A living catalog of every domain agent, which use cases each one owns, and which external tools each is allowed to call. When the router hands off a classified trigger, the registry resolves it to the right agent — or agents — and to the specific, scoped set of tools they're permitted to use for this task. Read-only access is enforced right here, at resolution time, as a second line of defense alongside whatever the network gateway already enforces.

Agent / use-case catalogTrigger-to-agent resolutionTool-scope enforcement
04

State / Workflow Manager

Execution & resilience

Every workflow the orchestrator runs is represented as a graph — nodes are agent or tool calls, edges are the control flow between them — with state that survives beyond any single request. If a step fails, it retries with backoff; if it hangs, it times out; if a domain agent is unavailable, the workflow degrades gracefully rather than failing outright. If the orchestrator itself restarts mid-workflow, execution resumes from where it left off.

Graph-based executionRetries & timeoutsDurable state across restartsGraceful degradation
05

Audit Logger

Governance

An append-only record of every decision the system makes: what came in, what confidence score it received, which tier it landed in, what action was taken, what the outcome was, who or what acted, and exactly when. Full coverage of every autonomous and human-approved action is a governance requirement, not an operational nicety — and it's the raw material the learning loop and any root-cause investigation both depend on.

Immutable decision trailFull coverage of acted decisionsFeeds learning & root-cause review
06

Memory & HITL Interface

Learning & the human layer

Two things live here. Memory reads relevant history and corrective-action playbooks into a workflow as it starts, and writes back what actually happened once it finishes, so the same mistake doesn't get made twice. HITL is the human side of the system: a weekly digest for anything that ran on its own, a short-turnaround approval queue for anything needing sign-off first, and a full-context escalation console for anything the system genuinely isn't confident about.

02 · How It Works

How a single request actually moves through the system

This is the mechanism, end to end — including the one deliberate shortcut (the real-time fast path) and the three-way fork every other result goes through before anything is allowed to happen.

1

A trigger arrives

A scheduled tick, a bus event, a threshold breach, or a direct request — all four land at the same ingress.

Real-time check

If this trigger belongs to one of the handful of use cases designated for the real-time fast path, the flow short-circuits here entirely: the responsible domain agent acts immediately, and the orchestrator is only notified afterward, asynchronously — purely so the action still gets audited. Everything else continues below.

2

Ingress classifies

Authenticates, normalizes the payload, and classifies both the intent and the priority of the request.

3

Registry resolves

Maps the classified trigger to the right domain agent(s) and the tool scopes they're allowed to use for this task.

4

Workflow instantiates

The graph for this run is created and relevant memory/context is loaded before execution starts.

5

The domain agent executes

Calls systems of record through scoped, read-only tool access, runs whatever model inference the task needs, and produces its output — a forecast, a plan, an alert, or a recommendation.

6

Confidence scores the result

The proposed action is scored and assigned to one of three tiers.

High confidence

Autonomous

Executes immediately. Shows up in the weekly digest for after-the-fact review.

Medium confidence

Copilot

Waits in the approval queue. A person confirms or rejects before it executes.

Low confidence

Escalate

Routes to a person with full context. Does not execute on its own.

7

Audit Logger writes the trail

Every run gets a complete record — inputs, score, tier, action, outcome — regardless of which path it took above.

8

Memory closes the loop

The outcome is stored, so future confidence scoring and routing get a little better every time.

03 · Build Roadmap

Phase-by-phase engineering plan

A foundation phase and a hardening phase bracket the three core build phases, each broken into concrete steps and exit criteria.

Phase 0 · Foundation

Decisions and scaffolding

1–2 wks

Steps

  • Resolve open decisions: event bus, API gateway, and its RBAC/encryption owner
  • Finalize the four canonical contracts as versioned schemas: Trigger, AgentInvocation, AgentResult, Decision
  • Stand up base environment: cluster namespace, secrets-vault scaffold, CI/CD, Dev → Test → Prod promotion path
  • Confirm status of the historical-data-platform migration as a tracked external dependency

Exit criteria

Ready for P1 whenContracts are frozen v1, environment is provisioned, and the event bus / gateway choices are locked.
Phase 1 · Data foundation & demand

Scheduled routing, registry, audit logging

3–5 wks

Steps

  • Trigger Router: scheduled-trigger path only — auth, validation, canonical envelope
  • Agent & Tool Registry against one external system first (the primary analytics platform) — catalog + resolution logic
  • State/Workflow Manager: graph skeleton, one linear flow, retries + timeouts, durable state across restarts
  • Audit Logger wired end to end from day one — inputs, confidence, tier, outcome
  • Confidence Engine shipped as a fixed-threshold stub (not calibrated yet)
  • Integration milestone: one scheduled forecast trigger runs end to end, dry-run

Exit criteria

Ready for P2 whenOne scheduled trigger runs reliably with a complete audit trail, and the registry correctly resolves against the analytics platform.
Phase 2 · Scheduling & real-time

Event-driven routing, fast-path, confidence live

4–6 wks

Steps

  • Event bus integration for the event-driven trigger type
  • Fast-path bypass for the real-time use cases: event → domain agent action directly, async audit notification back — needs coordination with the team building those agents
  • Confidence Decision Engine goes live: real evidence-based scoring replacing the fixed stub
  • Registry expands to the ticketing and CRM/forecast-input systems
  • HITL copilot tier: a time-boxed approval queue — queue, approve/reject, replay the approved action

Exit criteria

Ready for P3 whenFast-path measured under a minute end to end; confidence engine scoring real evidence, not a fixed threshold.
Phase 3 · Reporting & cross-domain

Threshold routing, peer reads, full HITL, learning loop

4–6 wks

Steps

  • Threshold trigger type support in the Ingress
  • Peer read-access wiring for the cross-domain agents (root-cause and executive-summary) across all domains
  • Memory & HITL completed: vector-store read/write, playbook store, weekly digest, escalation console with full context
  • Learning loop: confirmed outcomes feed back into confidence calibration on a weekly cadence
  • Observability fully wired — every decision logging confidence, inputs, and outcome
  • Validate non-functional targets: uptime, full audit coverage, model-cost budget

Exit criteria

Ready for hardening whenAll four trigger types are live, all HITL tiers are operational, and the learning loop demonstrably moves confidence calibration.
Phase 4 · Hardening

Calibration and resilience before calling it production

2–3 wks

Steps

  • Calibrate confidence thresholds against accumulated real audit outcomes
  • Resilience testing: orchestrator restart recovery, domain-agent outage survival
  • Security review: read-only scope enforcement, secrets audit, RBAC sign-off

Exit criteria

Calendar noteCalibration needs live traffic to accumulate outcomes — bounded by elapsed time, not headcount, so it doesn't compress much with a bigger team.

Risk factors — regardless of headcount

None of these are fixed by adding more engineers.

Historical-data-platform migration

A hard dependency for the whole mesh. If it slips, phases that depend on real historical data slip with it — an external dependency, not orchestrator dev time.

Open architectural decisions

Event bus, API gateway, and RBAC ownership still to be confirmed. Each week these stay open pushes Phase 0 — and everything after it — by a week.

Cross-team dependency on domain agents

The fast-path and the peer-mesh reads only work once the domain agents exist. If those teams run behind, the orchestrator's integration milestones wait.

Confidence calibration needs real traffic

Calibrating the autonomy thresholds against actual outcomes takes elapsed weeks of live decisions accumulating, not just engineering hours.
