Architecture case study · multi-agent systems

Agent Swarm for Workforce Management Operations

A hub-and-spoke orchestrator with a peer-mesh layer for cross-functional analysis: one orchestrator, five domain agents, two cross-cutting agents. Every anomaly is detected, diagnosed against a configurable root-cause knowledge base, and either auto-resolved, queued for a time-boxed human confirmation, or escalated — with a tamper-evident audit trail behind every decision.

01 · Topology

Hub-and-spoke core, peer mesh on top

The orchestrator is the only component that dispatches work — domain agents never call each other. Every agent result is published to a shared blackboard; two cross-cutting agents hold standing read access to it. Mesh visibility without mesh feedback loops.

02 · Decision routing

The confidence gate

Confidence is computed from evidence — data volume, tool failures, model agreement — not from a model's self-assessment alone. Every result passes through the gate; nothing reaches a system of record without it.

High

Autonomous

Corrective action executes and is logged automatically. Early in a rollout, execution can still be held for human confirmation regardless of score.

Medium

Copilot

Top-ranked causes with evidence land in an approval queue. A human confirms or overrides within a fixed SLA; approval replays the exact stored action.

Low

Escalate

Flagged as a novel case with the full evidence bundle. The outcome of the human investigation feeds the next classifier retraining cycle.

03 · Workflow

Breach to closed loop in eight steps

The diagnostic data flow, as implemented. Steps 1–2 are the trigger layer; 3–4 the evidence and classification layers; 5–8 routing, narrative, logging, and learning.

Monitor

Continuous KPI watch against per-domain thresholds.

Detect

A threshold breach becomes a trigger with a trace ID; breach details are audited before any work starts.

Collect

Domain agents pull evidence from every connected system in parallel and publish to the blackboard.

Classify

The root-cause agent scores the evidence against the configured knowledge base and ranks the top candidates with cited evidence.

Route

The confidence gate sends the result down the autonomous, copilot, or escalation path.

Generate

Structured narrative: breach summary, top cause, supporting causes, and the pre-approved corrective action.

Log

A tracking ticket is created with the full package; the decision and evidence land in the tamper-evident audit log.

Close the loop

Human confirmation is appended to a retraining feed; each case is stored as vector-searchable precedent for the next one.

04 · Knowledge base

Root causes are data, not model guesses

The classifier picks causes from a configurable taxonomy of failure categories per domain — the corrective action attached to a diagnosis is looked up from that taxonomy, never invented by the model. No match means novel cause → automatic escalation instead of a confident-sounding wrong answer.

A · Data & Input Failures

Upstream data quality issues that corrupt an output before any model runs — stale feeds, mapping mismatches, historical gaps.

B · Model & Methodology

Issues in the model itself — outdated patterns, missing event inputs, drift, mis-scoped assumptions.

C · External Drivers

Causes outside the system's control — demand surges, outages, unplanned business activity — detected and accounted for, not ignored.

D · Process & Governance

Timing, ownership, and handoff failures — late submissions, missed re-runs, mapping drift between systems.

05 · Fail-safe

What happens when something breaks

The swarm is designed around failure, not the happy path. Every degradation has a defined behavior — nothing hangs, nothing silently acts on bad evidence.

FailureBehavior
Preferred model unservable (host memory pressure)A config-driven fallback chain tries progressively smaller models, logs the degradation, and remembers what worked. Verified live: a 70B-parameter model unavailable, degraded to a 3B model, still produced a valid result.
Model host down / repeated failuresPer-model circuit breakers so one dead model can't stall every task; the result degrades to zero confidence → escalation.
Small model emits malformed structured outputSchema-constrained decoding forces valid structure even from a small fallback model, plus validation and one repair-retry.
Tool / integration errorRecorded in the result's evidence; confidence penalized; the decision and failure are audited.
Agent raises an exceptionIsolated to that one task — becomes an error result routed to escalation; the rest of the fan-out is unaffected.
Novel / off-taxonomy causeConfidence capped below the copilot band → always escalates to a human; logged to the retraining feed.
Unattended writesAll external writes are dry-run until explicitly enabled; even autonomous actions leave systems of record untouched by default.
Tampered historyHash-chained audit log — any edit breaks the chain, caught on verification.

06 · Scale

When the data is in the millions

The core principle: LLMs are the reasoning layer, never the data-processing layer. No raw rows ever enter a prompt — so LLM cost and latency stay flat no matter how large the source data grows.

millions of rows  →  deterministic aggregation  →  KB-sized evidence  →  LLM
   (warehouse / API)     (SQL, z-scores, Erlang-C)     (top-k, capped)    (1–2 calls)

A diagnostic prompt carries only a few KB of aggregated evidence — and that stays the same whether the source was 10K rows or 100M, because aggregation happens in code, before the model.

LayerSmall scaleMillions+
AggregationIn-agent stats over pullsPushed down to a warehouse / async analytics jobs
DetectionThreshold checks in codeSame math in a stream processor over live events — zero tokens per million events
ClassificationLLM over the taxonomyA trained classifier scores causes in milliseconds; the LLM writes the narrative only — call count per case stays 1
Vector memoryIn-process cosine scanAn ANN-indexed vector database
Stores & workersSingle-node store, one process/triggerA durable store · N stateless workers behind a queue
