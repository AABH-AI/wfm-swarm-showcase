Walkthrough · 1 of 6

The four front doors — where work comes from

Every piece of work enters through one of four doors — nobody has to remember to kick anything off.

TriggerPlain EnglishExample
Scheduled Calendar work — it starts because it's time, like payroll. Monday 06:00 — "build next week's call-volume forecast."
Event A source system pushed news the instant it happened. Genesys signals billing-queue volume just jumped above forecast.
Threshold A continuously watched number crossed an agreed line. Predicted service level for the next half-hour drops below 80/20.
Request A person — or another system — explicitly asked. A new ServiceNow ticket needs a priority and a resolver group.

Manual or autonomous at the start? Supervised first. Early on, every action is dry-run or human-reviewed. As the audit trail proves a use case right, it graduates: a human approves each action (copilot) → it acts alone with weekly review (autonomous). Autonomy is earned, never assumed.

Walkthrough · 2 of 6

The fork — what sends a trigger down the fast path?

One question decides it: is service being lost right now? The answer is fixed in advance by use case — never judged at runtime.

⚡ Fast path — three pre-registered use cases

Intraday re-optimization (UC7), live adherence alerts (UC10), breach prediction (UC11) — a queue is melting down now. Tagged realtime, recognized at the door, short-circuited: the agent acts first and reports after.

Orchestrator path — everything else

Forecasts, plans, schedules, summaries can afford minutes. Tagged standard or batch, they take the full pipeline: classify → route → score → (where needed) human approval before anything happens.

The routing parameters are two fields on every trigger: use case (UC7 / UC10 / UC11 = fast path) and priority tag (realtime / standard / batch). A fixed lookup — the fork can never guess wrong.

Walkthrough · 3 of 6

How big is the speed gap between the two paths?

Under a minute versus a few minutes — up to half an hour when a human approval is in the journey.

PathMachine timeWith human approvalWhy
⚡ Fast path< 1 minute end-to-end— no pre-approval gateStraight from Genesys to the acting agent. Reviewed after the fact.
Orchestrator pathSeconds of plumbing + minutes of agent work (< 5 min even for the heaviest analysis)+ up to 30 min in the approval queueThe real time is the agent's own work plus, for copilot actions, the queue.

That gap is why the fast path exists: a breach happening now can't wait five minutes — a weekly forecast can wait thirty.

Walkthrough · 4 of 6

Inside the orchestrator path — eight steps, each with one job

A useful mental model: a well-run office that never loses paperwork.

1

Ingress — the reception desk

Verifies the query / task, converts it to one standard form, tags it: which use case, how urgent.

2

Workflow manager — the project manager

Opens a durable case file: plans the steps, sets deadlines and retries, loads memory of similar past cases. A restart resumes the case — nothing is lost.

3

Registry — the company directory

Names the one domain agent that owns this work — and exactly which data it may read. Nothing more.

4

Domain agent — the specialist

Pulls the permitted data (read-only), runs its computation — formula, trained model, or LLM — and returns one output: a forecast, plan, alert, or recommendation. LLM drafts are validated first: a weak draft loops back for refinement before it counts.

5

Action routing — held at the gate

The proposal is back, but nothing executes yet: every action must first pass the trust check.

6

Confidence engine — the risk officer

Scores the proposal on evidence. High → executes now. Medium → human approval queue. Low → human console, no action. (How the score is built: part 6 below.)

7

Audit logger — the ledger

Records the whole story: input, proposal, score, who decided, outcome. Append-only — nobody edits history.

8

Memory update — the lessons-learned file

Stores the outcome so the next similar case starts smarter — and the use case earns future autonomy.

Walkthrough · 5 of 6

After the fork — and where ML, LLMs, and true agency live

Both roads end at the same two places — the audit ledger and the memory store. What differs is when the action happens and when humans look.

⚡ After the fast path

The action is already taken. An async audit event lands in the same ledger; the weekly digest reviews it after the fact and flags anything out of pattern.

After the orchestrator path

The output clears the confidence gate first — executed, approved-then-executed, or handed to a human. Same ending for all three: audit written, memory updated.

Pillar by pillar — formula, trained model, LLM, or agent?

ML appears where the past predicts the future; an LLM only where words are the product; full agency in exactly two places.

The headline: true agency exists in exactly two places — the real-time fast path and root-cause analysis. Everything else is a formula, a trained model, or one LLM call. Deliberate: cheap, fast, predictable, auditable.

Walkthrough · 6 of 6

The confidence score — and where people stay in the loop

A weighted blend of four evidence signals — a formula, not an opinion. Same evidence, same score: every tier decision can be re-derived and defended later.

Evidence signalThe question it answers
Model self-assessmentHow sure does the agent say it is?
Data quality & freshnessBuilt on complete, current data — or a stale feed?
Historical accuracy of this use caseHow often has it been right before? (From the audit trail.)
Deviation from the playbookIs the proposed action normal for this situation — or unusual?
≥ 85% · Autonomous

Humans review weekly

Executes immediately; reviewed in the weekly digest.

60–85% · Copilot

Human approves first — the main HITL gate

Waits in a 30-minute queue; only approval executes it. Every click trains calibration.

< 60% · Escalate

Human owns the decision

No action taken. Full evidence pack lands on a console; a person decides.

HITL is required at: (1) copilot actions — pre-execution yes/no; (2) escalations — humans decide outright; (3) the weekly digest — after-the-fact review of autonomous and fast-path actions; (4) calibration — human approve/reject clicks teach the thresholds.

03 · Build Roadmap

Phase-by-phase engineering plan

A foundation phase and a hardening phase bracket the three core build phases, each broken into concrete steps and exit criteria.

Phase 0 · Foundation

Decisions and scaffolding

1–2 wks

Steps

  • Resolve open decisions: event bus, API gateway, and its RBAC/encryption owner
  • Finalize the four canonical contracts as versioned schemas: Trigger, AgentInvocation, AgentResult, Decision
  • Stand up base environment: cluster namespace, secrets-vault scaffold, CI/CD, Dev → Test → Prod promotion path
  • Confirm status of the historical-data-platform migration as a tracked external dependency

Exit criteria

Ready for P1 whenContracts are frozen v1, environment is provisioned, and the event bus / gateway choices are locked.
Phase 1 · Data foundation & demand

Scheduled routing, registry, audit logging

3–5 wks

Steps

  • Trigger Router: scheduled-trigger path only — auth, validation, canonical envelope
  • Agent & Tool Registry against one external system first (the primary analytics platform) — catalog + resolution logic
  • State/Workflow Manager: graph skeleton, one linear flow, retries + timeouts, durable state across restarts
  • Audit Logger wired end to end from day one — inputs, confidence, tier, outcome
  • Confidence Engine shipped as a fixed-threshold stub (not calibrated yet)
  • Integration milestone: one scheduled forecast trigger runs end to end, dry-run

Exit criteria

Ready for P2 whenOne scheduled trigger runs reliably with a complete audit trail, and the registry correctly resolves against the analytics platform.
Phase 2 · Scheduling & real-time

Event-driven routing, fast-path, confidence live

4–6 wks

Steps

  • Event bus integration for the event-driven trigger type
  • Fast-path bypass for the real-time use cases: event → domain agent action directly, async audit notification back — needs coordination with the team building those agents
  • Confidence Decision Engine goes live: real evidence-based scoring replacing the fixed stub
  • Registry expands to the ticketing and CRM/forecast-input systems
  • HITL copilot tier: a time-boxed approval queue — queue, approve/reject, replay the approved action

Exit criteria

Ready for P3 whenFast-path measured under a minute end to end; confidence engine scoring real evidence, not a fixed threshold.
Phase 3 · Reporting & cross-domain

Threshold routing, peer reads, full HITL, learning loop

4–6 wks

Steps

  • Threshold trigger type support in the Ingress
  • Peer read-access wiring for the cross-domain agents (root-cause and executive-summary) across all domains
  • Memory & HITL completed: vector-store read/write, playbook store, weekly digest, escalation console with full context
  • Learning loop: confirmed outcomes feed back into confidence calibration on a weekly cadence
  • Observability fully wired — every decision logging confidence, inputs, and outcome
  • Validate non-functional targets: uptime, full audit coverage, model-cost budget

Exit criteria

Ready for hardening whenAll four trigger types are live, all HITL tiers are operational, and the learning loop demonstrably moves confidence calibration.
Phase 4 · Hardening

Calibration and resilience before calling it production

2–3 wks

Steps

  • Calibrate confidence thresholds against accumulated real audit outcomes
  • Resilience testing: orchestrator restart recovery, domain-agent outage survival
  • Security review: read-only scope enforcement, secrets audit, RBAC sign-off

Exit criteria

Calendar noteCalibration needs live traffic to accumulate outcomes — bounded by elapsed time, not headcount, so it doesn't compress much with a bigger team.

Risk factors — regardless of headcount

None of these are fixed by adding more engineers.

Historical-data-platform migration

A hard dependency for the whole mesh. If it slips, phases that depend on real historical data slip with it — an external dependency, not orchestrator dev time.

Open architectural decisions

Event bus, API gateway, and RBAC ownership still to be confirmed. Each week these stay open pushes Phase 0 — and everything after it — by a week.

Cross-team dependency on domain agents

The fast-path and the peer-mesh reads only work once the domain agents exist. If those teams run behind, the orchestrator's integration milestones wait.

Confidence calibration needs real traffic

Calibrating the autonomy thresholds against actual outcomes takes elapsed weeks of live decisions accumulating, not just engineering hours.
